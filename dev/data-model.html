TellingTree — Data Model

How the Database Fits Together

TellingTree's database is built around one central idea: everything belongs to a tree. A tree is a family unit — it contains persons, their relationships, the stories told about them, media files, and tags for organizing it all. Users own and collaborate on trees.

The Core Loop

User ──creates──> Tree ──contains──> Persons ──connected by──> Relationships
                   │                    │
                   │                    └──featured in──> Stories ──organized by──> Tags
                   │                                        │
                   └──stores──────────> Media <──attached to─┘

A user registers, creates a tree, adds persons (grandma, uncle, etc.), defines relationships between them (parent, spouse, mentor), and writes stories that reference those persons. Media (photos, audio, video) can be attached to stories or persons. Tags help organize stories by theme (childhood, holidays, war years).

Tables in Detail

users

The account table. Holds login credentials and profile info.

Column Type Purpose
id UUID Primary key
email VARCHAR(255) Login identifier, validated format
username VARCHAR(100) Display handle, alphanumeric + dash/underscore
password_hash VARCHAR(255) Bcrypt hash, never stored in plaintext
full_name VARCHAR(255) Optional display name
is_active BOOLEAN False = account deactivated
token_version INTEGER Incremented on password change to revoke all tokens
created_at TIMESTAMPTZ
updated_at TIMESTAMPTZ Auto-updated

Key behaviors:

  • A user can own multiple trees and be a member of others' trees
  • Deleting a user requires transferring all owned trees first
  • token_version is embedded in JWTs — changing password invalidates all sessions

trees

The top-level container. Everything else hangs off a tree.

Column Type Purpose
id UUID Primary key
owner_id UUID FK → users. The creator/owner
name VARCHAR(255) "The Johnson Family", etc.
description TEXT Optional longer description
is_public BOOLEAN If true, anyone can view (no login needed for reads)
created_at TIMESTAMPTZ
updated_at TIMESTAMPTZ

Key behaviors:

  • Owner has full control (delete, transfer, manage members)
  • Ownership can be transferred to a member via PUT /trees/{id}/transfer
  • Deleting a tree cascades to all its persons, stories, relationships, media, and tags

tree_members

The sharing/collaboration table. Links users to trees with a role.

Column Type Purpose
id UUID Primary key
tree_id UUID FK → trees
user_id UUID FK → users
role VARCHAR(20) viewer, editor, or admin
created_at TIMESTAMPTZ

Constraint: UNIQUE(tree_id, user_id) — a user can only have one role per tree.

Role hierarchy:

viewer  → can read everything in the tree
editor  → viewer + create/update/delete persons, stories, relationships, media, tags
admin   → editor + manage members (invite, change roles, remove)
owner   → admin + delete tree, transfer ownership (not stored here — it's tree.owner_id)

The owner is not in this table. They're above the member system entirely.

persons

An individual in the family tree. This is a genealogy record, not a user account — persons don't log in.

Column Type Purpose
id UUID Primary key
tree_id UUID FK → trees. Scopes the person to a tree
given_name VARCHAR(255) First name
family_name VARCHAR(255) Last name / surname
birth_date DATE
death_date DATE
birth_location VARCHAR(255)
gender VARCHAR(50) Free-text, not an enum
bio TEXT Short biography
created_at TIMESTAMPTZ
updated_at TIMESTAMPTZ

Key behaviors:

  • All fields except tree_id are optional — you might only know a name, or only a photo
  • gender is stored as free text for maximum flexibility
  • Deleting a person cascades to their relationships, story links, and media

relationships

A typed connection between two persons. This is what makes the "tree" a tree.

Column Type Purpose
id UUID Primary key
tree_id UUID FK → trees
person_a_id UUID FK → persons. The "from" side
person_b_id UUID FK → persons. The "to" side
relationship_type VARCHAR(100) Free text: "parent", "spouse", "godparent", "mentor", etc.
direction VARCHAR(20) one_way or mutual
start_date DATE When the relationship began (e.g., marriage date)
end_date DATE When it ended (e.g., divorce date)
notes TEXT
created_at TIMESTAMPTZ
updated_at TIMESTAMPTZ

Constraint: CHECK(person_a_id != person_b_id) — can't relate someone to themselves.

How direction works:

  • mutual: "A and B are spouses" — the type applies both ways
  • one_way: "A is parent of B" — the type reads from A to B only

Why free-text types? An enum would force "parent/child/spouse/sibling" and miss godparents, mentors, caretakers, best friends, step-parents, etc. The frontend can suggest common types while allowing custom input.

stories

The core storytelling unit — a narrative about one or more persons.

Column Type Purpose
id UUID Primary key
tree_id UUID FK → trees
title VARCHAR(255) "Summer of '62", "The Immigration", etc.
content TEXT The story body (plain text or markdown)
event_date DATE When the story happened
event_end_date DATE For stories spanning a range
event_location VARCHAR(255) Where it happened
author_id UUID FK → users. Who wrote this story
created_at TIMESTAMPTZ
updated_at TIMESTAMPTZ

Key behaviors:

  • Stories are linked to persons via story_persons (many-to-many)
  • Stories are tagged via story_tags (many-to-many)
  • Media can be attached to stories
  • Deleting a story cascades to its person links, tag links, and media

story_persons

Junction table linking stories to the persons featured in them.

Column Type Purpose
id UUID Primary key
story_id UUID FK → stories
person_id UUID FK → persons

Constraint: UNIQUE(story_id, person_id)

This enables queries like "show me all stories about Grandma" or "who appears in this story?"

media

Uploaded files — photos, audio clips, videos, documents.

Column Type Purpose
id UUID Primary key
tree_id UUID FK → trees
story_id UUID FK → stories (nullable). Attaches media to a story
person_id UUID FK → persons (nullable). Attaches media to a person (e.g., profile photo)
uploaded_by_id UUID FK → users
filename VARCHAR(255) UUID-based filename on disk
original_filename VARCHAR(255) What the user uploaded ("grandma_1962.jpg")
mime_type VARCHAR(100) "image/jpeg", "audio/mp3", etc.
size_bytes BIGINT
storage_path VARCHAR(500) Relative path: {tree_id}/{media_id}.ext
media_type VARCHAR(20) photo, audio, video, document, or other
caption TEXT Optional description
created_at TIMESTAMPTZ

Key behaviors:

  • storage_path is relative — resolved at runtime via resolve_path() in app/services/storage.py
  • Files are stored at storage/media/{tree_id}/{media_id}.ext
  • FK cascade: deleting a story or person deletes its media rows, and an after_delete event cleans up the files from disk
  • MIME types are validated against an allowlist before upload

tags

Labels for organizing stories within a tree.

Column Type Purpose
id UUID Primary key
tree_id UUID FK → trees
name VARCHAR(100) "childhood", "war years", "holidays", etc.
color VARCHAR(7) Hex color for UI display ("#FF5733")
created_at TIMESTAMPTZ

Constraint: UNIQUE(tree_id, name) — tag names are unique within a tree.

story_tags

Junction table linking stories to tags.

Column Type Purpose
id UUID Primary key
story_id UUID FK → stories
tag_id UUID FK → tags

Constraint: UNIQUE(story_id, tag_id)

Cascade Rules Summary

When you delete something, here's what happens:

Deleted Cascades to
Tree All persons, relationships, stories, media (files included), tags, members
Person Their relationships, story_person links, media (files included)
Story story_person links, story_tag links, media (files included)
Tag story_tag links
User tree_memberships (must transfer owned trees first)

Indexes

Index Purpose
users(email), users(username) Fast login lookups
persons(tree_id) List persons by tree
relationships(tree_id), relationships(person_a_id), relationships(person_b_id) Relationship queries
stories(tree_id), stories(author_id) Story listing and filtering
media(tree_id), media(story_id), media(person_id) Media lookups
tags(tree_id) Tag listing

Design Principles

  1. UUIDs everywhere — all primary keys. URL-safe, no information leakage.
  2. Tree-scoped — every piece of data belongs to exactly one tree. This makes access control simple: check tree permissions once, then all data within is accessible.
  3. Flexible by default — relationship types, gender, and tags are all free-text. No rigid enums that would need schema changes to extend.
  4. Cascade carefully — deleting a tree cleans up everything. Deleting a person cleans up their relationships and media. No orphaned data.
  5. Relative storage paths — media paths in the DB are relative, resolved at runtime. This makes the DB portable across storage backends.
  6. Timestamps are timezone-aware — all TIMESTAMPTZ, always UTC.
